# NT5 Miniport Support for Win9x Backport

## Context

The current shim (NTMINI.VXD) loads NT4-era ScsiPort miniport drivers inside Win98. The project is named "NT5-9x Driver Backport" but does NOT actually support NT5 drivers yet.

Key discovery: NT5 (Windows 2000/XP) completely redesigned the IDE driver stack. NT5's atapi.sys is NOT a ScsiPort miniport. It is a standalone WDM driver that communicates with pciidex.sys via an internal IRB (IDE Request Block) protocol and imports heavily from ntoskrnl.exe (PnP, Power, object manager, DPCs, timers, spinlocks). Shimming NT5 atapi.sys directly would require reimplementing a significant portion of the NT I/O manager. NOT practical.

However, NT5 still has ScsiPort-based miniport drivers for non-IDE controllers (Symbios, Adaptec, BusLogic, etc.). These use the same ScsiPort API with one key addition: HwScsiAdapterControl callback for PnP/power management.

## Three Viable Paths

### Path A: Extend ScsiPort shim for NT5 ScsiPort miniports (RECOMMENDED FIRST)

NT5 ScsiPort miniports use an extended HW_INITIALIZATION_DATA structure. The difference:
- NT4: HwInitializationDataSize = 0x40 (no HwScsiAdapterControl)
- NT5: HwInitializationDataSize = 0x44 (adds HwScsiAdapterControl at end)

NT5's scsiport.sys checks this size field. If the miniport provides HwScsiAdapterControl, scsiport calls it during power transitions (ScsiStopAdapter, ScsiRestartAdapter, ScsiSetBootConfig, ScsiSetRunningConfig).

Our shim can:
1. Check the size field when the miniport calls ScsiPortInitialize
2. If NT5-extended, extract HwScsiAdapterControl pointer
3. Call it with ScsiSetRunningConfig during init (the one that matters for non-PnP)
4. Ignore the power-related calls (Win98 doesn't do PnP power transitions)

This is a bounded addition. Most NT5 ScsiPort miniports will work with just this change, since the core ScsiPort API is stable. Some may fail if they depend on features our shim doesn't provide (WMI, SRB extensions), but that's discoverable per-driver.

Target drivers: sym_hi.sys (Symbios/LSI), aic78xx.sys (Adaptec), buslogic.sys (BusLogic), any NT5 ScsiPort miniport that still uses the classic ScsiPort model.

### Path B: Load NT5's scsiport.sys as intermediary (AMBITIOUS)

Instead of shimming ScsiPort ourselves, load NT5's actual scsiport.sys binary. It speaks the correct ScsiPort miniport API on one side; we shim its ntoskrnl.exe imports on the other.

NT5 scsiport.sys imports from ntoskrnl.exe and HAL.dll. Required shim surface:
- IoXxx: IoCallDriver, IoCompleteRequest, IoAllocateIrp, IoFreeIrp, IoGetCurrentIrpStackLocation, IoAllocateErrorLogEntry, IoAttachDeviceToDeviceStack, IoRegisterDeviceInterface (can stub PnP-related ones)
- KeXxx: KeInitializeSpinLock, KeAcquireSpinLock, KeReleaseSpinLock, KeInitializeDpc, KeInitializeTimer, KeSetTimer, KeInsertQueueDpc (need real ring-0 implementations using VMM)
- ExXxx: ExAllocatePoolWithTag, ExFreePoolWithTag (map to VxD heap)
- PnP/Power: Mostly stubs (PoStartNextPowerIrp, PoCallDriver, IRP_MJ_PNP dispatch)
- Registry: ZwOpenKey, ZwQueryValueKey, ZwClose (map to VMM registry services)
- HAL: READ_PORT_UCHAR, WRITE_PORT_UCHAR, etc. (direct port I/O, trivial)

This is more work but has a huge payoff: any NT5 ScsiPort miniport loads without modification, and ScsiPort handles all the miniport management we currently do manually.

Estimated: 40-60 ntoskrnl functions to shim, maybe 20 of which need real implementations (the rest can be stubs or no-ops for the non-PnP Win98 context).

### Path C: NT4 miniports cover IDE already (FALLBACK)

NT4's atapi.sys handles IDE/ATAPI hardware perfectly. The hardware hasn't changed. For IDE controllers, there is no practical reason to use NT5's atapi.sys.

NT5 support matters for SCSI controllers where vendor miniports may only exist in NT5 form (newer Adaptec, LSI, etc.). Path A handles this.

### Path D: Full WDM shim for NT5 atapi.sys + pciidex.sys (HARDEST, HIGHEST PAYOFF)

Load NT5's native IDE stack: pciidex.sys + pciide.sys + atapi.sys. This requires a WDM compatibility layer on top of Win98's VMM that provides:

- Full IRP infrastructure: allocation, completion, stack locations, forwarding
- PnP manager: AddDevice, device object creation/attachment, device tree
- Power manager: PoXxx functions, power IRP dispatch
- Object manager: device objects, driver objects, reference counting
- DPC/Timer/Spinlock: real ring-0 implementations via VMM primitives
- Registry: ZwOpenKey, ZwQueryValueKey, ZwSetValueKey via VMM registry
- Memory manager: MmMapIoSpace, MmGetPhysicalAddress, pool allocation
- IRB protocol between atapi.sys and pciidex.sys (internal, must reverse-engineer or trace)
- PCI bus driver simulation: enumerate IDE controllers, provide config space

This is effectively a mini NT kernel running inside a VxD. Estimated 80-120 ntoskrnl/HAL functions, of which 40-50 need real implementations. The rest can be stubs for non-PnP context.

The payoff: any WDM storage driver from NT5 runs on Win98. Not just IDE, but USB mass storage (usbstor.sys), IEEE 1394 storage, and future driver architectures. This is the path that makes the project name honest.

Realistic scope: massive, but tractable in phases. Start with the minimum set for atapi.sys DriverEntry to complete, then incrementally add functions as the driver exercises new code paths.

## Implementation Plan

### Phase 1: HwScsiAdapterControl support (Path A)

1. Read NT5 ScsiPort miniport documentation for HwScsiAdapterControl
   - Acceptance: understand all four control types and which matter for non-PnP

2. Modify ScsiPortInitialize shim in NTMINI_V5.C:
   - Check HwInitializationDataSize field
   - If >= 0x44, extract HwScsiAdapterControl from extended structure
   - Store in global state alongside HwFindAdapter, HwStartIo, etc.
   - Acceptance: correctly detects NT4 vs NT5 miniport

3. Call HwScsiAdapterControl(ScsiSetRunningConfig) after HwInitialize
   - This tells the miniport "you're in a running system, configure accordingly"
   - Acceptance: NT5 miniport receives the call and doesn't crash

4. Test with an NT5 ScsiPort miniport binary
   - Obtain sym_hi.sys or aic78xx.sys from Windows 2000
   - Embed as byte array (same pattern as atapi.sys)
   - If import resolution fails, identify missing ScsiPort exports and add them
   - Acceptance: NT5 miniport loads, DriverEntry succeeds

5. Update documentation and repo name/description to accurately reflect NT4+NT5 ScsiPort support
   - Acceptance: README, ABOUT, STATUS.md all accurate

### Phase 2: ntoskrnl shim for NT5 scsiport.sys (Path B, future)

6. Dump NT5 scsiport.sys import table
   - Catalog every ntoskrnl.exe and HAL.dll import
   - Classify: needs real impl / can stub / can no-op
   - Acceptance: complete import list with classifications

7. Implement core ntoskrnl shim functions
   - Spinlocks: map to VMM critical sections or cli/sti
   - Pool allocation: map to VxD_HeapAllocate / _PageAllocate
   - DPC/Timer: map to VMM Set_Global_Time_Out or similar
   - IRP infrastructure: minimal IRP allocation and completion
   - Acceptance: shim compiles and provides all required exports

8. PE-load NT5 scsiport.sys alongside a miniport
   - Load scsiport.sys first, resolve its imports against our ntoskrnl shim
   - Load miniport, resolve its imports against scsiport.sys exports
   - Call scsiport.sys DriverEntry, which should initialize and call miniport
   - Acceptance: both binaries load, DriverEntry chain completes

9. Bridge scsiport.sys's output to IOS
   - scsiport.sys creates device objects and handles IRPs
   - We intercept at the device object level and route to/from IOS
   - Acceptance: IOS sees the device, drive letter assigned

Phase 1 and Phase 2 code written (2026-03-28). Phase 3 code written. Remaining: multi-DLL PE loader, integration testing with actual NT5 binaries.

### Phase 3: WDM compatibility layer for NT5 native drivers (Path D)

10. Reverse-engineer NT5 atapi.sys import table
    - Dump all imports from ntoskrnl.exe and HAL.dll
    - Do the same for pciidex.sys and pciide.sys
    - Merge into master list, classify each: real impl / stub / no-op
    - Acceptance: complete import catalog with effort estimates per function

11. Implement IRP infrastructure
    - IoAllocateIrp, IoFreeIrp, IoCallDriver, IoCompleteRequest
    - IRP stack locations (IoGetCurrentIrpStackLocation, IoGetNextIrpStackLocation)
    - IoSetCompletionRoutine, I/O completion callbacks
    - Device objects and driver objects (IoCreateDevice, IoDeleteDevice, IoAttachDeviceToDeviceStack)
    - Acceptance: can allocate an IRP, push it through a driver stack, and complete it

12. Implement kernel primitives on VMM
    - KeInitializeSpinLock / KeAcquireSpinLock / KeReleaseSpinLock: cli/sti or VMM critical sections
    - KeInitializeDpc / KeInsertQueueDpc: map to VMM Schedule_Global_Event or similar
    - KeInitializeTimer / KeSetTimer: map to VMM Set_Global_Time_Out
    - KeInitializeEvent / KeSetEvent / KeWaitForSingleObject: VMM semaphore primitives
    - ExAllocatePoolWithTag / ExFreePoolWithTag: VxD_HeapAllocate or _PageAllocate
    - Acceptance: kernel primitives work in ring-0 VxD context

13. Implement minimal PnP manager
    - AddDevice callback dispatch
    - IRP_MJ_PNP with IRP_MN_START_DEVICE, IRP_MN_QUERY_CAPABILITIES (minimum set)
    - Stub remaining PnP minors (REMOVE, STOP, SURPRISE_REMOVAL, etc.)
    - Fabricate resource lists (I/O ports, IRQs) from known hardware config
    - Acceptance: atapi.sys AddDevice completes, START_DEVICE IRP succeeds

14. Implement minimal Power manager
    - PoStartNextPowerIrp, PoCallDriver, PoRequestPowerIrp
    - IRP_MJ_POWER with IRP_MN_SET_POWER (report D0/S0 always)
    - Acceptance: power IRPs don't crash, driver reaches working state

15. Implement PCI bus simulation
    - Enumerate IDE controllers via direct PCI config space scan (0xCF8/0xCFC)
    - Create PDO (Physical Device Object) for each IDE channel
    - Provide BUS_INTERFACE_STANDARD for config space reads
    - Acceptance: pciidex.sys enumerates channels, creates FDOs

16. Load the NT5 IDE stack
    - PE-load pciidex.sys, resolve against ntoskrnl shim
    - PE-load pciide.sys, resolve against pciidex.sys exports (PciIdeXInitialize)
    - PE-load atapi.sys, resolve against ntoskrnl shim
    - Trigger PnP enumeration: pciidex creates channel PDOs, atapi attaches as channel driver
    - Acceptance: all three binaries loaded, device stack assembled

17. Bridge NT5 device objects to IOS
    - Intercept IRPs at the top of the NT5 device stack
    - Translate IOS IORs into IRPs, send down the stack
    - Translate completed IRPs back to IOR status codes
    - Acceptance: IOS sees the device, IORs flow through NT5 stack, data returns

18. End-to-end validation
    - Boot Win98 with full NT5 IDE stack loaded via VxD
    - Read sectors from CD-ROM through the NT5 path
    - Compare with NT4 atapi.sys path for correctness
    - Acceptance: ISO 9660 CD001 found, data matches, no corruption
